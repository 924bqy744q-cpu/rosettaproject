01 — Research

What we studied.
What we took. What we rejected.

Six products represent the best-in-class UI thinking for different aspects of Rosetta. This is what we learned from each.

Linear
Project tool · linear.app

The gold standard for "functional beauty." Dark-mode-first, razor-sharp typography, zero wasted pixels. Every element earns its place.

✓ LCH color space for perceptually even tones
✓ Dense-but-clear information hierarchy
✓ Keyboard-first, then mouse — every action is fast
✗ Too task-heavy for a conversational learning flow
Perplexity
AI search · perplexity.ai

Proved that an AI interface can feel like a search box. One job, done with clarity. The head of design said: "It's a simple design problem — sources and an answer."

✓ Single-focused input as the entire home screen
✓ No navigation clutter on first use
✓ Sources surfaced immediately for trust
✗ Lacks warmth — too clinical for learning
Duolingo
Learning · duolingo.com

Deferred account creation — users learn before they sign up. The product proves its value before asking for commitment. A masterclass in onboarding psychology.

✓ Value before commitment — no sign-up wall
✓ Progress indicators as motivation, not progress
✓ Mobile-first, thumb-zone aware
✗ Gamification distracts from depth of understanding
Cleo
Finance app · meetcleo.com

Conversational onboarding done right. Casual, relatable questions that feel like talking to a friend, not filling out a form. Personality and warmth baked into the UX.

✓ Questions feel like dialogue, not data collection
✓ One question at a time — never a form
✓ Typing indicators make the AI feel alive
✗ Too playful — Rosetta needs intellectual gravitas
Notion
Productivity · notion.so

The best example of progressive disclosure — complexity hidden until needed. Users feel powerful without being overwhelmed. Minimal chrome, maximum content.

✓ Generous whitespace creates focus
✓ Typography as the primary design element
✓ Context-sensitive UI — nothing shown before needed
✗ Too open-ended — Rosetta needs more guidance
Slack
Messaging · slack.com

Onboarding uses the product's own core mechanic — messaging — to teach itself. The empty state is a first message. This is our model for how the explanation should feel like a conversation.

✓ Uses the product metaphor for onboarding
✓ "Tada!" — emotional welcome that signals personality
✓ Sequential reveal — one thing at a time
✗ Complex navigation patterns don't apply here

02 — Principles

Six rules that
govern every decision.

Before opening Figma, these principles must be internalized. Every pixel is accountable to them.

01

One thing at a time

Each screen has exactly one primary action. No competing calls to action. The user should never wonder what to do next — the answer should be visually obvious. Working memory has 4 slots. Don't fill them with UI.

02

The explanation is the hero

Chrome is a supporting cast. Navigation, branding, and chrome should recede. When the explanation appears, nothing should compete with it visually. The interface becomes almost invisible during the learning moment.

03

Warmth through typography, not decoration

No illustrations. No stock photos. No decorative icons. Personality comes from typeface choice, weight, and rhythm. The serif display font carries the warmth. The monospace carries precision. Together they signal: intelligent and human.

04

Sequential revelation

Explanation chunks appear one at a time, not all at once. The interface controls pacing deliberately — not as a gimmick, but because dumping all three chunks simultaneously recreates the cognitive overload we're trying to solve.

05

Loading is a feature

The 3-5 second generation wait is not a problem to hide — it's a trust-builder. "Building your explanation..." signals that something real is being constructed for you specifically, not retrieved from a shelf. Never use a generic spinner alone.

06

Mobile-first, touch-first

Design at 390px width first. Every interactive element must be at minimum 44×44px touch target. Primary actions sit in the bottom 40% of screen (thumb zone). Desktop is an enhancement, not the primary canvas.

06 — Rules

Do this.
Never that.

✓ Always
  • One primary action per screen
  • Progress dots, not percentage bars
  • DM Serif for emotional moments (the click, the headline)
  • Left-border accent on explanation chunks
  • Bottom-anchored input on mobile
  • Show subsumer name during loading
  • 44px minimum touch targets
  • Use the ✦ sparingly — only on success
  • Warm neutral backgrounds (#F7F6F2), never pure white
  • Surface the "not clicked" state with kindness, not failure language
✗ Never
  • Navigation bar on learning screens
  • Show all explanation chunks at once
  • Use sliders for state capture — use segmented buttons
  • Generic spinner without context text
  • Use Inter, Roboto, or system fonts
  • Purple gradients (overused AI aesthetic)
  • Illustrations or stock imagery
  • More than 2 CTAs on any screen
  • Sign-up wall before the magic moment
  • Percentage-based completion bars (gamification creep)

07 — Figma Setup

How to build this
in Figma, step by step.

1

Create the file structure

Create a new Figma file. Name it "Rosetta — Design System & Screens." Create 4 pages:

🎨 Design Tokens 🧩 Components 📱 Mobile Screens 🖥️ Desktop Screens
2

Install fonts

Before doing anything else, install all three fonts via Google Fonts or Figma's font picker. Verify they render correctly at 14px body size.

DM Serif Display DM Sans DM Mono
3

Set up color styles (Design Tokens page)

Create every color as a named Figma Style. Use the exact naming convention below — your developer will reference these names in CSS variables.

ink/100 ink/60 ink/30 ink/10 surface/white surface/100 surface/200 surface/300 accent/terracotta accent/forest accent/amber
4

Set up text styles

Create named text styles for every typographic role. Never apply font settings directly to elements — always via style.

display/hero (DM Serif, 48px, -2%) display/h1 (DM Serif, 36px, -2%) display/h2 (DM Serif, 24px, -1%) body/lg (DM Sans, 18px, 400) body/md (DM Sans, 15px, 400) body/sm (DM Sans, 13px, 400) label/md (DM Mono, 11px, 500, +12%) label/sm (DM Mono, 10px, 500, +15%)
5

Build the component library (Components page)

Build these as Figma components with all interactive states. Use Auto Layout on every component — no fixed-size elements.

Button/Primary Button/Secondary Button/Ghost Input/Default Input/Active Input/Textarea ChatBubble/System ChatBubble/User TypingIndicator ExplanationChunk ProgressDots SegmentedControl/3 ResultCard/Success ResultCard/Partial ResultCard/Retry LoadingState
6

Build the 7 mobile screens (Mobile Screens page)

Use iPhone 14 Pro frame (390 × 844px) for all mobile screens. Build each of the 7 session states as a separate frame. Arrange them left-to-right in order with 80px gaps. Add a flow connector (Figma prototype) between each.

For each screen, use components from step 5. Never place raw elements directly — always use component instances.

7

Build the desktop screens (Desktop Screens page)

Desktop frame: 1440px wide. Same 7 states but centered content column at max-width 640px. The session flow stays single-column — desktop is not a two-column layout. Add a subtle background texture (very low opacity noise, or a warm gradient from surface/100 → surface/200).

8

Create the prototype flow

Connect all 7 screens in sequence using Figma Prototype. Use "Smart Animate" transitions with 280ms ease-out for all screen changes. Add interaction on the Continue button, chat send action, and CTA buttons. Share the prototype link with the dev team.

9

Export specs for developer handoff

Use Figma's Dev Mode to annotate spacing and sizes. Make sure every component has:

All states documented (hover, focus, disabled) Padding/spacing annotated Color style named (not hex) Motion spec in description field
